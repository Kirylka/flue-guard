# Writing principles for these docs

Working guide for anyone editing `docs/`. Not a site page (dotfile — VitePress
skips it). Distilled from: the Diátaxis framework (diataxis.fr, all four
quadrant pages), Divio's "Grand Unified Theory of Documentation"
(documentation.divio.com), Moesif's and Mintlify's teardowns of why Stripe's
docs are the gold standard, and Write the Docs' beginner's guide.

## Structure

1. **Every page belongs to exactly one Diátaxis quadrant** — tutorial, how-to,
   reference, or explanation — and the sidebar says which. Mixing quadrants on
   one page is the root cause of bad docs (Diátaxis/Divio).
2. **README order is fixed**: one-sentence problem → minimal working example →
   install → docs links. A stranger must know what the project is for in
   ten seconds (Write the Docs).
3. **How-to titles start with the reader's task** ("Make retries safe"), one
   intent per page. How-tos assume competence and never teach concepts.
4. **Reference mirrors the module structure** — one section per entry point,
   the same names as the code. It describes; it never instructs or persuades.
5. **Explanation pages answer one "why" each.** They are the only home for
   opinions, trade-offs, and rejected alternatives.
6. **No FAQ page.** A recurring question is a docs bug — fix the page it
   should have been answered on.

## Code

7. **The tutorial succeeds in under 5 minutes** from `npm install` to a
   visibly governed call: a denied unauthorized call and a verified audit
   line, printed. Every step produces visible output; the tutorial offers
   **zero choices** (one runtime, one path — alternatives live in how-tos).
8. **Every code block is copy-paste runnable as-is**: imports included, no
   `// ...` elisions, stand-ins declared inline with `declare const`. All
   `ts` blocks are extracted and compiled against the real `src/` by
   `npm run docs:check-samples`; a block that can't compile doesn't ship.
9. **Code before prose.** Show `govern()` working before describing it; teach
   every concept with a runnable example before naming its abstraction.
10. **Reference is verified against the real exports** (the compile check +
    konsistent's export conventions), so it cannot drift. Never document what
    doesn't exist.

## Security docs specifically

11. **Guarantees and non-guarantees are stated together** (the trust-model
    page): "gates run before `execute`" next to "the library cannot bound a
    primitive's payload". Every stated guarantee has a corresponding test.
12. **Errors are entry points.** Readers arrive from stack traces: every
    thrown error name and every `GovernanceErrorCode` appears verbatim on the
    errors reference page.
13. **Facts that must stay correct** (re-verify on Flue upgrades): Flue
    beta.9 `input`/`run` tool contract; Node >= 22.19; peer
    `@flue/runtime >=1.0.0-beta.9 <1.0.0`; Valibot `parameters` forwarded to
    Flue, anything else degrades model schema guidance to an unconstrained
    object; handler results must be JSON-plain; idempotency keys and
    requested scopes are audited unredacted.
